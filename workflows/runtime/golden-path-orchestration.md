Golden Path Orchestration

Status: draft runtime architecture
Format: store-os runtime spec
Version: 1.0
Phase: Phase 11 — Runtime / Execution Layer

IMPORTANT — DRAFT GOLDEN PATH NOTE:
This document describes a planned golden-path execution chain. It is not a record of an
actual execution. No store has been processed through this chain. No Shopify data has been
imported. No artifacts have been generated by a live system. This is a concrete, step-by-step
specification of the intended minimum viable execution flow for a single store — from intake
through publish decision — written to reduce ambiguity for the next implementation step.

—

Purpose

This document defines the minimal runnable execution chain for a single store from intake
through publish decision. It is the "golden path" — the simplest, most direct path through
the system with no errors, no retries, and all artifacts produced in order.

The golden path is intentionally simplified:
  - Sequential execution only (no parallelism in Phase 11 initial build)
  - Single product type (simple store with products and collections)
  - English-language, single-market store
  - manual_review_required: true (conservative default)
  - All planning artifacts produced (no skipped workflows)

The golden path is the baseline against which errors, partial success, and parallel
optimizations are measured.

—

Pre-Execution Checklist (Human Operator)

Before starting the chain, a human operator must confirm:

□ n8n instance is running and accessible
□ STORE_OS_OPENAI_CREDENTIAL_NAME credential exists in n8n credential store
□ STORE_OS_SHOPIFY_CREDENTIAL_NAME credential exists in n8n credential store
□ STORE_OS_ENV is set (dev | staging | production)
□ STORE_OS_PROJECT_ROOT exists and is writable
□ STORE_OS_LLM_MODEL is set
□ STORE_OS_SHOPIFY_API_VERSION is set to a current supported version
□ STORE_OS_AJV_SCHEMA_PATH points to the schemas/ directory from this repository
□ All 34 n8n workflows have been imported and are in ACTIVE status
□ The orchestrate-runtime-execution workflow is the entry point

Golden Path Execution Sequence

The following is the step-by-step execution sequence for one store.
Each step shows: the workflow invoked, its key inputs, its output artifact, and the
expected duration estimate (rough, for planning only — actual times will vary).

—

Step 0 — resolve-runtime-config
Trigger: Manual (operator starts the chain)
Key inputs: STORE_OS_* environment variables
Output: runtime-config payload (in-memory only)
Next: orchestrate-runtime-execution
Duration estimate: seconds
Notes: Validates all credentials exist. If anything fails here, the chain does not start.

—

Step 1 — orchestrate-runtime-execution
Trigger: Called by resolve-runtime-config
Key inputs: runtime-config payload
Output: orchestration controller (manages all subsequent steps)
Duration: spans the full run

—

Step 2 — intake-store-input
Trigger: orchestrator triggers intake; operator fills the n8n form
Key inputs (operator supplies):
  project_id: "example-store-2025"
  project_name: "Example Store"
  primary_market: "United States"
  primary_language: "en"
  primary_currency: "USD"
  vertical: "consumer-goods"
  price_positioning: "mid-market"
  manual_review_required: true
  publish_mode: "draft"
  shopify_store_selector: "example-store.myshopify.com"
  (plus any optional fields: competitor hints, SEO preferences, motion level, etc.)
Output: normalized intake payload + projects/example-store-2025/ folder created
Artifact written: none (folder creation only at this step)
Next: import-shopify-data
Duration estimate: depends on operator form-fill time

—

Step 3 — import-shopify-data
Trigger: orchestrator calls after intake
Key inputs: shopify_store_selector from intake, SHOPIFY_CREDENTIAL
Shopify API calls made:
  GET /admin/api/{version}/shop.json
  GET /admin/api/{version}/products.json (paginated)
  GET /admin/api/{version}/collections.json (smart, paginated)
  GET /admin/api/{version}/custom_collections.json (paginated)
  GET /admin/api/{version}/collects.json (paginated)
Output artifact: projects/example-store-2025/shopify-import.json
Next: build-store-profile
Duration estimate: 30 seconds – 5 minutes (depends on store size)
Notes: shopify-import.json is a raw data snapshot. It is NOT schema-validated
  against any planning artifact schema. It is treated as operational input only.

HUMAN REVIEW GATE (if manual_review_required: true)
Trigger: orchestrator reaches review gate after all Phase 1–9 artifacts are produced
Action: n8n Wait node pauses chain, human is notified
Human reviews: all artifacts produced so far (30 artifacts)
Human may:
  - Approve continuation (POST to webhook → chain resumes at run-validation)
  - Request re-run of specific workflows (manually trigger those workflows, then approve)
  - Reject (chain does not proceed — artifacts remain in project folder for later use)
Duration: up to human decision time

Golden Path Duration Estimate

Rough total time estimate for a small store (< 100 products, < 20 collections):
  Steps 0–2 (setup + intake): operator-dependent
  Steps 3–28 (import + planning chain): 20–60 minutes of LLM synthesis time
  Human review gate: operator-dependent
  Steps 29–33 (governance + backup): 5–15 minutes

Total wall-clock time: 30–90 minutes (excluding human review wait time)

This estimate is rough. Actual times depend on:
  - Store size (number of products, collections)
  - LLM model latency
  - OpenAI API rate limits
  - n8n execution overhead
  - File I/O speed

Golden Path Deviation Scenarios

Deviation 1 — Shopify import fails
  Chain halts at Step 3.
  Human fixes credential or store selector.
  Chain resumes from Step 3.

Deviation 2 — Schema validation fails for a planning artifact
  Chain halts at the affected step.
  Artifact is quarantined.
  Human reviews quarantine artifact, identifies schema violation.
  LLM prompt is adjusted. Workflow is re-run.
  Chain resumes from that step.

Deviation 3 — Validation report is FAIL
  Chain continues to build-publish-decision.
  publish-decision.json is BLOCKED.
  Human reviews validation-report.json, identifies critical failures.
  Human chooses which upstream workflows to re-run.
  After fixes: human re-triggers from run-validation.

Deviation 4 — Human review gate: operator rejects
  Chain does not proceed.
  Artifacts remain in project folder.
  Operator may manually re-run specific workflows.
  Or choose to archive the project and start fresh.

Open Questions

1. How does the human reviewer access the planning artifacts for the review gate?
   A separate n8n workflow that reads and renders the artifacts? A file browser?
   A simple HTML report? This is the UX question for the review gate.

2. Should the golden path include an automated pre-validation before the human review gate?
   A lightweight presence and schema check before the full run-validation would surface
   obvious issues earlier.

3. Should the chain log estimated completion times? This would help operators plan
   the review gate timing.

4. What is the definition of "approval" at the review gate? An n8n form submission?
   A Slack approve button? An email reply? This must be specified before implementing
   the Wait node.
