# Artifact Model

## Artifact Identity

Every artifact in store-os has a unique identity encoded in the `_meta.artifact_id` field:

```
Format: {project_id}/{artifact_name}/{run_id}
Example: suppliedtech/store-profile/run-20260323142301-a4f2
```

The `artifact_id` is globally unique per run. If a phase is re-run (e.g., retried), the new run produces a new `artifact_id`.

---

## Schema Versioning

All artifact schemas carry a `schema_version` field in their `_meta` block. The version format is `MAJOR.MINOR` (e.g., `1.0`).

Version rules:
- Minor version bumps (`1.0` → `1.1`): additive changes only (new optional fields). All existing consumers remain compatible.
- Major version bumps (`1.x` → `2.0`): reserved for breaking changes (required field added, existing field removed/renamed). These require migration.

Current schema versions: all Phase 1 artifacts are at `1.0`.

---

## Artifact Metadata (`_meta` field)

All artifact schemas support an optional `_meta` field. When populated, it contains:

```json
{
  "_meta": {
    "artifact_id": "suppliedtech/store-profile/run-20260323-a4f2",
    "schema_version": "1.0",
    "phase_id": "build-store-profile",
    "run_id": "run-20260323142301-a4f2",
    "created_at": "2026-03-23T14:23:01.000Z",
    "upstream_dependencies": [
      { "artifact_name": "shopify-import", "phase_id": "import-shopify-data" },
      { "artifact_name": "normalized_intake_payload", "phase_id": "intake-store-input" }
    ],
    "provenance": "shopify-derived",
    "confidence": "high",
    "cloud_mode": true,
    "persistence_mode": "cloud-inline"
  }
}
```

The `_meta` field is optional. Artifacts without it are still valid — the envelope carries the same identity information in `envelope.store` and `envelope.runtime`.

---

## Upstream Dependency Tracking

The `_meta.upstream_dependencies` array records which artifacts were consumed to produce this artifact. Each entry specifies:

- `artifact_name`: the artifact file name (e.g., `shopify-import`, `store-profile`)
- `phase_id`: which phase produced it
- `artifact_id` (optional): full artifact ID if available

This enables forward and backward lineage tracing across the chain.

---

## Provenance Types

The `_meta.provenance` field classifies how an artifact was produced:

| Value | Meaning |
|---|---|
| `llm-synthesized` | Entirely generated by LLM from internal signals. Requires human review. |
| `direct-from-intake` | Directly mapped from user intake input. High fidelity. |
| `shopify-derived` | Derived from Shopify API data. Accuracy depends on store data quality. |
| `derived-heuristic` | Computed by deterministic code heuristics (e.g., catalog_type from product count). |
| `mixed` | Combines multiple provenance types. |

Use `provenance` to communicate how much trust to place in artifact values.

---

## Confidence Levels

The `_meta.confidence` field rates overall artifact confidence:

| Value | Meaning |
|---|---|
| `high` | Strong signals from intake or direct data. Low uncertainty. |
| `medium` | Reasonable signals but some inference required. |
| `low` | Mostly inferred with limited signal. Human review important. |
| `unknown` | Provenance or signal quality is unclear. |

For LLM-synthesized artifacts, confidence is typically `low` to `medium`. The artifact's `notes` field and the `open_questions` array (store-profile) provide more granular uncertainty signals.

---

## Persistence Modes

The `_meta.persistence_mode` field describes how the artifact was stored:

| Value | Meaning |
|---|---|
| `disk` | Written to disk in the project directory (self-hosted mode). |
| `inline-only` | Never persisted to disk. Lives only in the n8n pipeline memory (cloud mode). |
| `cloud-inline` | n8n Cloud mode: artifact is passed inline between workflow steps. |

In self-hosted mode, artifacts are written to `{project_root}/{project_id}/{artifact_name}.json`.

---

## Cloud vs Self-Hosted Artifact Lifecycle

### Cloud Mode (n8n Cloud)

1. Artifact is assembled in memory
2. Inline validation runs (enum checks, required fields)
3. AJV schema validation is skipped (no local schema files accessible)
4. Artifact is NOT written to disk
5. Artifact is forwarded inline in the n8n pipeline to the next node
6. `persistence_mode: 'cloud-inline'`

### Self-Hosted Mode

1. Artifact is assembled in memory
2. Inline validation runs
3. AJV schema validation runs against the local schema file
4. On validation failure: artifact is quarantined (not written to final path)
5. On validation success: atomic write via `{artifact}.tmp.json` → rename
6. Checkpoint and execution log are updated
7. `persistence_mode: 'disk'`

---

## Quarantine Behavior

When AJV validation fails in self-hosted mode, the artifact is quarantined rather than written to the final path:

```
Quarantine path: {project_root}/{project_id}/.runtime/quarantine/{artifact-name}.{timestamp}.rejected.json
```

The quarantine file contains:
- `quarantine_reason`: always `SCHEMA_VIOLATION`
- `artifact_name`: the artifact type
- `workflow_id`: which workflow produced it
- `validation_errors`: the AJV error list
- `artifact_payload`: the rejected artifact for inspection

Quarantine is non-destructive. The previous valid artifact (if any) remains in place. The chain is halted by throwing `SCHEMA_VIOLATION`.

## Schema Extension Rules

Artifact schemas follow additive-only extension rules:

1. **Never remove a required field** — this is a breaking change
2. **Never rename an existing field** — this is a breaking change
3. **New fields must be optional** — do not add them to the `required` array
4. **New enum values** on existing optional fields are acceptable
5. **New enum values** on existing required fields are only acceptable if all consumers handle unknown values gracefully

The `additionalProperties: false` constraint on most schemas means new optional fields must be explicitly added to the schema's `properties` section. The AJV validation will reject artifacts with properties not listed in the schema.

---

## Extension Guidance

To add a new optional field to an existing artifact schema:

1. Add the field to the schema's `properties` section with its type and description
2. Do NOT add it to the `required` array
3. If the schema uses `additionalProperties: false`, adding to `properties` is sufficient
4. Bump the schema's comment version in the file header (no formal version field in most schemas)
5. Update the relevant contract file (`workflows/contracts/{phase}.contract.json`) under `new_optional_fields`
6. Update the workflow code to populate the new field (conditionally — never throw if the field cannot be computed)
7. Update the gold path example if the field is expected in smoke tests
